1. Introduction et Objectifs

Ce travail pratique a pour objectif de développer un service REST complet en Java, permettant d'effectuer des opérations arithmétiques de base (addition, soustraction, multiplication, division). Le projet illustre les principes fondamentaux de l'architecture REST et l'utilisation de JAX-RS (Jakarta RESTful Web Services) avec Jersey comme implémentation.

Objectifs pédagogiques

  • Comprendre la conception d'une API REST selon les principes du cours
  • Créer un service REST en Java avec JAX-RS
  • Utiliser Jersey comme implémentation JAX-RS
  • Structurer un projet Maven avec packaging WAR
  • Déployer le projet dans Apache Tomcat
  • Manipuler les annotations REST (@Path, @GET, @QueryParam, @Produces, etc.)
  • Tester les endpoints avec Postman

3. Architecture du Projet

3.1 Architecture REST

L'architecture suit le modèle REST (Representational State Transfer) qui repose sur le protocole HTTP. Le schéma ci-dessous illustre le flux de communication entre le client et le serveur :

Figure 1 : Architecture REST - Flux Client → Tomcat → Jersey → Ressource

3.2 Flux de traitement d'une requête

  1. Client (Postman/Navigateur) : Envoie une requête HTTP GET avec des paramètres
  2. Apache Tomcat : Reçoit la requête et la transmet au conteneur servlet
  3. Jersey (JAX-RS) : Analyse l'URL, identifie la ressource et la méthode correspondante
  4. CalculRest (Ressource) : Traite la requête, appelle la logique métier
  5. Calculatrice (Logique métier) : Effectue le calcul
  6. Réponse JSON : Le résultat est sérialisé en JSON et renvoyé au client

3.3 Pattern MVC adapté

Le projet suit une architecture en couches :

  • Couche Présentation (REST) : CalculRest.java - Gère les requêtes HTTP
  • Couche Métier : Calculatrice.java - Contient la logique de calcul
  • Couche Modèle : Operation.java - Enumération des opérations
  • Configuration : RestApplication.java - Point d'entrée JAX-RS

5. Implémentation des Classes

5.1 Classe Calculatrice.java

Cette classe contient la logique métier pure, sans aucune dépendance aux technologies REST. Elle implémente les quatre opérations arithmétiques de base. 

public class Calculatrice {
    public double addition(double a, double b) {
        return a + b;
    }
    
    public double soustraction(double a, double b) {
        return a - b;
    }
    
    public double multiplication(double a, double b) {
        return a * b;
    }
    
    public double division(double a, double b) {
        if (b == 0) {
            throw new ArithmeticException("Division par zéro impossible");
        }
        return a / b;
    }
}

5.2 Enumération Operation.java

Définit les types d'opérations supportées avec une méthode de conversion depuis une chaîne. 

public enum Operation {
    ADDITION("Addition"),
    SOUSTRACTION("Soustraction"),
    MULTIPLICATION("Multiplication"),
    DIVISION("Division");
    
    private final String description;
    
    public static Operation fromString(String op) {
        return Operation.valueOf(op.toUpperCase());
    }
}

5.3 Ressource REST CalculRest.java

Cette classe expose les endpoints REST. Elle utilise les annotations JAX-RS pour définir les routes et les méthodes HTTP.

Annotations principales utilisées :

  • @Path("/calculatrice") : Définit le chemin de base de la ressource
  • @GET : Indique que la méthode répond aux requêtes HTTP GET
  • @Produces(MediaType.APPLICATION_JSON) : Format de réponse JSON
  • @QueryParam("a") : Récupère le paramètre 'a' de l'URL
  • @DefaultValue("0") : Valeur par défaut si le paramètre est absent

5.4 Configuration RestApplication.java

Point d'entrée de l'application JAX-RS qui enregistre les ressources REST. 

@ApplicationPath("/api")
public class RestApplication extends Application {
    @Override
    public Set<Class<?>> getClasses() {
        Set<Class<?>> classes = new HashSet<>();
        classes.add(CalculRest.class);
        return classes;
    }
}

6. Configuration JAX-RS

6.1 Fichier web.xml

Le fichier web.xml configure le servlet Jersey pour intercepter toutes les requêtes vers /api/*. 

<servlet>
    <servlet-name>Jersey REST Service</servlet-name>
    <servlet-class>org.glassfish.jersey.servlet.ServletContainer</servlet-class>
    <init-param>
        <param-name>jakarta.ws.rs.Application</param-name>
        <param-value>com.example.rest.RestApplication</param-value>
    </init-param>
    <load-on-startup>1</load-on-startup>
</servlet>

<servlet-mapping>
    <servlet-name>Jersey REST Service</servlet-name>
    <url-pattern>/api/*</url-pattern>
</servlet-mapping>

7.2 Déploiement sur Tomcat

Deux méthodes de déploiement sont possibles :

Méthode 1 : Déploiement manuel

  1. Copier le fichier calculatrice-rest.war dans le dossier webapps/ de Tomcat
  2. Démarrer Tomcat : bin/startup.bat (Windows) ou bin/startup.sh (Linux)
  3. Tomcat décompresse automatiquement le WAR
  4. L'application est accessible à : http://localhost:8080/calculatrice-rest/

Méthode 2 : Via le Manager de Tomcat

  1. Accéder à http://localhost:8080/manager/html
  2. Utiliser la section "WAR file to deploy"
  3. Sélectionner le fichier calculatrice-rest.war
  4. Cliquer sur "Deploy"

7.3 Vérification du déploiement

9. Fonctionnement de l'API

9.1 Principe REST

L'API respecte les principes REST (Representational State Transfer) :

  • Stateless : Chaque requête est indépendante, aucune session n'est maintenue
  • Ressources identifiées par URI : Chaque opération a son propre chemin
  • Méthode HTTP GET : Utilisée pour récupérer des résultats (opérations idempotentes)
  • Format JSON : Représentation standardisée des données
  • Codes HTTP : 200 OK pour succès, 400 Bad Request pour erreurs

9.2 Format de réponse uniforme

Toutes les réponses suivent la même structure :

Champ Type Description
operation String Type d'opération effectuée
operandeA Double Premier opérande
operandeB Double Deuxième opérande
resultat Double Résultat du calcul
message String Message de succès ou d'erreur
succes Boolean Indicateur de succès de l'opération

9.3 Avantages de cette architecture

  • Séparation des préoccupations : Logique métier isolée de la couche REST
  • Réutilisabilité : La classe Calculatrice peut être utilisée ailleurs
  • Testabilité : Chaque couche peut être testée indépendamment
  • Maintenabilité : Code organisé et facile à modifier
  • Extensibilité : Ajout facile de nouvelles opérations
  • Interopérabilité : Format JSON compatible avec tous les clients

9.4 Sécurité et bonnes pratiques

  • Validation des paramètres d'entrée
  • Gestion des exceptions (division par zéro, paramètres invalides)
  • Réponses structurées avec codes HTTP appropriés
  • Messages d'erreur explicites pour le débogage
